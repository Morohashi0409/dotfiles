# /pre-review

`/pre-review` コマンドが実行された場合、**自動コードレビューアシスタント**としての役割を果たしてください。以下のステップに従って、コミット前にコード品質を向上させます。

## 1. コンテキスト収集と実行
1.  **ステータス確認**: `git status` を実行して変更されたファイルを特定します。ステージングまたは変更がない場合は、その旨をユーザーに伝えて終了します。
2.  **レビューツールの実行**: 以下のコマンドを実行して、未コミットの変更に対するAIフィードバックを取得します。
    ```bash
    coderabbit --prompt-only -t uncommitted
    ```
    *（注: `coderabbit` コマンドが見つからない場合は、`cr` を試すか、正しいコマンドをユーザーに確認してください。）*
    *（注: 解析には時間がかかります（約180秒）。コマンドがバックグラウンドに移行した場合、完了まで諦めずにポーリング（sleep）を継続してください。）*

## 2. 分析と分類
レビューツールの出力結果を分析します。*各指摘事項*について、以下の2つのカテゴリのいずれかに分類します。

### カテゴリ A: 自動修正可能（確度が高い）
*   **タイプミス**: コメント、変数名（内部的なもの）、文字列リテラルのスペルミス。
*   **構文/フォーマット**: セミコロンの欠落、インデントの問題、末尾のカンマ、または明確な修正方法が1つだけの単純な構文エラー。
*   **ドキュメント**: コードから意図が明らかな場合のJSDoc/コメントの追加。
*   **安全なリファクタリング**: 単純なコードの簡略化（未使用変数の削除、`var` から `const/let` への変換など）。

### カテゴリ B: 協議が必要（人間の判断が必要）
*   **ロジック変更**: 制御フロー、ビジネスロジック、またはアルゴリズムを変更する提案。
*   **アーキテクチャ変更**: コードの移動、新規ファイルの作成、デザインパターンの変更などの提案。
*   **曖昧な修正**: 複数の有効な解決策が存在する問題。
*   **セキュリティ/パフォーマンス**: 慎重な検討が必要な潜在的な脆弱性やボトルネックに関する警告。
*   **破壊的変更**: 公開APIやインターフェースへの変更。

## 3. 実行フェーズ
1.  **自動修正の適用**:
    - **カテゴリ A** のすべての項目について、即座に修正をファイルに適用します。
    - **検証**: `git diff` などを実行し、適用した修正が正しいか、新たな問題を引き起こしていないかを確認します。
2.  **自動修正のコミット**:
    - 修正が適用された場合、**`commit.md` のルール（Prefix形式など）に従い、必ず日本語で**コミットメッセージを作成し、コミットしてください。
    - 例: `git commit -m "【💄: style】 coderabbitレビューによる自動修正を適用"`
    - *注:* カテゴリ A の項目がない場合は、このステップをスキップします。

## 4. 報告フェーズ（ユーザーへの出力）
結果を以下のMarkdown形式で提示してください：

```markdown
# 🔍 プレレビューサマリー

## ✅ 自動修正してコミットしました
*（カテゴリ A から修正・コミットされた項目をリストアップしてください。ない場合は「なし」と記載してください。）*
- [File.ts] コメントのタイプミス修正: "recieve" -> "receive"
- [Config.json] 末尾のカンマを追加しました。

## ❓ 確認が必要です
*（カテゴリ B の項目をリストアップしてください。各項目について、ファイル名、問題点、およびユーザーへの質問/選択肢を提示してください。）*
1. **[File.ts] Nullポインタの可能性**
   - *指摘事項:* レビューアが `userData` のNullチェックを追加することを提案しています。
   - *質問:* ガード節 `if (!userData) return;` を追加しますか？
   
2. **[API.ts] パフォーマンス警告**
   - *指摘事項:* `processData` 内でネストされたループが検出されました。
   - *質問:* Map検索を使用するようにリファクタリングしますか？

*（最後に、「確認が必要です」の項目についてどのように進めるか質問してください。）*
```

## 5. 対話的フォローアップ
- ユーザーが「確認が必要です」の項目に対して回答した場合、その決定に従って実装を行い、それらの変更に対して**別のコミット**を作成してください（例：`git commit -m "refactor: address code review feedback"`）。
