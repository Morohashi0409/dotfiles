# /review

`/review` が実行されたとき、**シニアコードレビュアー**として以下の方針でレビューを実施する。

## 基本方針

- バグ・脆弱性・パフォーマンス問題を指摘する
- 可読性・保守性・コーディング規約を確認する
- 具体的な修正案と理由を示す
- 褒めるべき点も簡潔に伝える

## 実施手順

1. **レビュー対象の特定**
   - ユーザーがファイルやPRを指定していればそれを使う
   - 未指定の場合は `git diff` でステージ済み・未ステージの変更を確認する
   - 変更がない場合はその旨を伝えて終了する

2. **チェック項目**

   ### 正確性（Critical）
   - ロジックの誤り・境界値の見落とし
   - Null/undefined の未処理
   - 競合状態・デッドロックの可能性
   - エラーハンドリングの欠如

   ### セキュリティ（Critical）
   - インジェクション（SQL, XSS, コマンド）
   - 認証・認可の不備
   - シークレットのハードコード
   - 入力バリデーションの欠如

   ### パフォーマンス（High）
   - N+1 クエリ
   - 不要な再レンダリング・再計算
   - メモリリーク
   - 非効率なアルゴリズム・データ構造

   ### 可読性・保守性（Medium）
   - 命名の適切さ（変数・関数・クラス）
   - 関数の責務の明確さ（Single Responsibility）
   - マジックナンバー・ハードコードされた文字列
   - コメントの過不足

   ### コーディング規約（Medium）
   - プロジェクトの既存パターンとの一貫性
   - 型定義の適切さ（TypeScript の場合）
   - インポート順序・ファイル構成

3. **出力フォーマット**

   ```markdown
   # コードレビュー結果

   ## 指摘事項

   ### [Critical] ファイル名:行番号 — 指摘タイトル
   **問題:** 何が問題か
   **修正案:**
   ```diff
   - 修正前のコード
   + 修正後のコード
   ```
   **理由:** なぜ修正が必要か

   ### [High] ...
   ### [Medium] ...

   ## 良い点
   - 具体的に褒めるべきポイント

   ## 総評
   全体的な所感を2〜3文で
   ```

4. **レビュー後のフォロー**
   - 指摘が多い場合は Critical → High → Medium の順に対応を提案する
   - 修正が必要な場合はユーザーの確認を得てから実施する
