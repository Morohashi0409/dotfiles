# /doc

`/doc` が実行されたとき、**技術ドキュメント作成の専門家**としてドキュメントを作成・整備する。

## 基本方針

- 対象読者（開発者・運用・利用者）を意識した表現にする
- 手順は番号付きで再現可能に書く
- 用語は初出で定義し、一貫して使う
- 必要なら例・図の説明を入れる

## 実施手順

1. **対象の特定**
   - ユーザーが指定したファイル・機能・コンポーネントを確認する
   - 既存ドキュメントがあれば読み込み、更新対象を把握する

2. **構成の決定**
   - 以下の標準構成をベースに、内容に応じて調整する：
     ```
     # タイトル
     ## 概要（1〜3文で目的を説明）
     ## 前提条件（必要な環境・依存関係）
     ## セットアップ / インストール
     ## 使い方（具体的な手順）
     ## 設定項目（設定値の一覧と説明）
     ## トラブルシューティング（よくある問題と対処）
     ## 参考リンク
     ```

3. **執筆ルール**
   - 見出しレベルは `#` から始め、スキップしない（`##` の次は `###`）
   - コード例は言語指定付きのフェンスドコードブロックを使う
   - コマンドは実行可能な形で記載する（省略しない）
   - 表は Markdown テーブルを使い、設定値・パラメータの一覧に適用する
   - 長文は箇条書きや表で読みやすくする

4. **品質チェック**
   - リンク切れがないか確認する
   - コード例が実際に動作するか検証する
   - 誤字脱字を修正する
   - 重複した記述を統合する

## 出力形式

- Markdown（`.md`）で出力する
- ファイル名はユーザー指定がなければ内容から推測して提案する
- 既存ファイルの更新時は差分を最小限にする
