あなたはプロエンジニアです。以下の方針でcommit, pushを実施してください。

## コミットメッセージの基本方針

- **commitメッセージには何をしたかを具体的に書く**
- 未来の自分やチームメイトが履歴を見たときに、何をしたか一目でわかるようにする
- 「バグ修正」「ドキュメント更新」など曖昧な表現は避け、**具体的な変更内容**を記載する

## Prefix形式の使用

コミットメッセージの冒頭には **Prefix** をつける。形式は以下：

```
【{emoji}: {type}】 {subject}
```

### 主なPrefix一覧

| Prefix | 説明 | 使用例 |
|--------|------|--------|
| `🐛 fix` | バグ修正 | 【🐛: fix】 ユーザー名が正しく表示されないバグを修正 |
| `🎸 feat` | 新機能追加 | 【🎸: feat】 プロフィール画像アップロード機能を追加 |
| `✏️ docs` | ドキュメント更新 | 【✏️: docs】 READMEにセットアップ手順を追記 |
| `💡 refactor` | コードリファクタ | 【💡: refactor】 認証処理をユーティリティに分離 |
| `💍 test` | テスト追加 | 【💍: test】 ログインAPIのユニットテストを追加 |
| `💄 style` | コードの書式変更 | 【💄: style】 ESLintのフォーマットを適用 |
| `⚡️ perf` | パフォーマンス改善 | 【⚡️: perf】 画像読み込みを遅延ロードに変更 |
| `🤖 chore` | ビルド・設定など | 【🤖: chore】 依存パッケージを更新 |
| `⚙️ ci` | CI設定・スクリプト | 【⚙️: ci】 GitHub Actionsのワークフローを追加 |
| `🚧 wip` | 作業中のコミット | 【🚧: wip】 検索機能の実装途中 |

### 記載ルール

- **Prefix部分**: コミットの種類を表すラベル（変更内容に応じて適切なものを選択）
- **subject部分**: 具体的な変更内容を簡潔に記載（64文字以内推奨）

## 良い例・悪い例

### ❌ 避けるべき例

```
一覧画面でのNullポインタ例外を修正
ログインユーザー認証機能を追加
READMEに環境構築手順を追加
```
→ カテゴリが一目でわからず、履歴を探しづらい

### ✅ 推奨する例

```
【🐛: fix】 一覧画面でのNullポインタ例外を修正
【🎸: feat】 ログインユーザー認証機能を追加
【✏️: docs】 READMEに環境構築手順を追加
```
→ 種類と内容が一目でわかり、履歴が追いやすい

## 実施手順

1. 変更内容に応じて適切なPrefixを選択する
2. 具体的な変更内容を簡潔に記載する（何を・なぜ・どこを変更したか）
3. `git commit` または `git cz` でコミット
4. `git push` でリモートにプッシュ
