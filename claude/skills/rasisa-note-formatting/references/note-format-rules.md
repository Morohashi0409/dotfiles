# note Format Rules For RASHISA

This reference collects the note-specific editorial and formatting heuristics used by `rasisa-note-formatting`.

## Source Notes

### 1. sumini — note投稿向けSkillの整理
Source:
- https://note.com/sumini/n/ne85df1da8866

Useful takeaways:
- raw Markdown is not enough for reliable note posting workflows
- image lines, captions, embeds, and quote sources should be handled deliberately
- inline code is a weak fit for note and often needs conversion
- article text and asset operations should be separated

### 2. マナミ — noteタイトルの型
Source:
- https://note.com/mana2518/n/nc891ff5c720f

Useful takeaways:
- titles should reveal the value instead of hiding it
- three useful patterns:
  - conclusion-first
  - specific / concrete
  - two-sentence summary
- important keywords should appear early
- longer titles are acceptable if that improves clarity
- do not inflate expectations beyond the body

### 3. フクロウ — noteの読まれ方
Source:
- https://note.com/smart_logic/n/n06d0acf9ffb2

Useful takeaways:
- readers decide quickly on mobile
- title + first 2-3 lines do most of the entry work
- title, structure, line breaks, image placement, and ending all affect reading
- the article itself is often the first contact before the profile

## Practical Rules

### Title

- start with the article’s main keyword
- tell the reader what they will get
- prefer clarity over mystery
- use concrete words, numbers, or reader situations
- keep it honest

### Opening

- first 2-3 lines should stand on their own in feed/search previews
- surface the reader’s pain, curiosity, or benefit quickly
- avoid diary-like warmup unless that is the whole point of the article

### Structure

- one article = one theme
- one section = one message
- use short mobile-friendly paragraphs
- headings should still make sense if skimmed alone

### Visuals

- use a cover image with a clear relation to the theme
- add inline images only when they explain, prove, or reset attention
- prefer 1 image every 2-4 sections, not every paragraph
- captions should explain why the image is there

### Links And Quotes

- standalone URLs are better for embed candidates than markdown inline links
- inline markdown links can be converted to readable text + URL when needed
- quote sources should be tracked explicitly outside body prose

## RASHISA Angle Defaults

Strong note topics for this repo:

- one MBTI / love / strengths / 360 type at a time
- one hidden-strength story
- one misunderstanding or “why it gets misread” angle
- one workplace or self-analysis problem
- one clear service value proposition

Weak topics:

- broad template dumping
- vague “personality is interesting” articles
- service explanation mixed with several unrelated type explanations
